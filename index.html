Flagship Project 03

SAR Data Processing Element

PM · Architecture 설계

SAR 프로세싱 전 과정을 DAG로 구성·자동화하는 처리 시스템 (MSA · EDA)

Problem

기존에는 SAR 처리의 각 단계를 사람이 하나씩 수작업으로 실행했습니다. 담당자가 명령을 순서대로 돌리고 중간 산출물을 확인하며 다음 단계로 넘기는 방식이라 느리고 휴먼에러에 취약했습니다. 게다가 단계마다 대용량 산출물을 NAS에 쓰고 다시 읽어 I/O 대기가 반복됐고, 단일 처리로 한 장면에 10분 이상이 걸렸습니다.

Solution

사용자가 DAG 화면에서 처리 단계를 직접 구성하면 전 과정이 자동 실행되도록 설계. 단계 간 데이터를 NAS 왕복 대신 메모리 로딩으로 전달해 I/O 병목을 제거하고, 영상을 Azimuth 방향 블록으로 분할해 GPU 병렬 처리하도록 알고리즘 실행 구조를 재설계했습니다. 처리 단계를 독립 서비스로 나눈 MSA와 이벤트 큐 기반 EDA로 확장·재시도가 가능한 구조를 확립했습니다.

사람이 단계마다 명령을 돌리던 수작업 처리를, DAG로 정의한 파이프라인을 시스템이 스스로 순서대로 실행하는 무인 자동화로 전환했습니다. 담당자가 붙어 있지 않아도 수집→처리→후처리가 끝까지 흐르고, 실패 단계는 이벤트 큐에서 재시도됩니다.

Deep Dive · 처리 구조 재설계

병목의 실체는 알고리즘이 아니라 NAS 쓰기·읽기 대기였습니다. 단계 간 중간 산출물을 파일로 떨구지 않고 메모리에 올린 채로 넘기도록 실행 모델을 바꿔 I/O를 걷어내고, 그 위에서 영상을 Azimuth 방향 블록으로 잘라 GPU에 병렬 투입해 처리 자체를 스케일아웃했습니다.

Role

PM으로 요구사항·일정·이해관계자를 관리하면서, 아키텍처 설계(MSA 서비스 경계, 이벤트 기반 처리 흐름, DAG 오케스트레이션 모델, 성능 개선 설계)를 직접 담당했습니다. 현재 진행 중인 프로젝트입니다.

Flagship Project 04

LIID · 위성영상 정보 배포 서비스

PM · Architecture 설계

SDPE 산출물을 사용자 서비스로 전달 · LLM + DEM 기반 3D 지형 모니터링 · SAR 영상 검색 · 촬영 요청

Problem

SDPE가 만들어낸 production 영상과 정보는 그 자체로는 사용자가 쓸 수 있는 서비스가 아닙니다. 지형을 3D로 보여주려면 DEM과 영상을 합친 대용량 데이터를 그대로 내려줘야 해 응답이 느려지고, 영상 검색·촬영 요청·지형 분석이 서로 다른 경로로 흩어져 있었습니다. 위성 데이터 특성상 분석에 외부 LLM API로 데이터를 내보낼 수도 없었습니다.

Solution

SDPE 산출물을 받아 서비스 도메인으로 변환·제공하는 백엔드를 설계하고, 사용자에게는 LLM + DEM 기반 3D 지형 모니터링과 SAR 영상 검색·촬영 요청을 하나의 진입점으로 제공합니다. LLM은 로컬(온프레미스) 모델로 구성해 데이터를 밖으로 내보내지 않고, 대용량 3D 전달 경로에는 캐시 서버를 두어 Hit/Miss 전략으로 응답 지연을 흡수했습니다.

캐시 흐름

첫 요청(Miss)만 원본까지 내려가고, 이후 동일 영역 요청은 캐시에서 끝납니다. 관심 지역이 반복 조회되는 모니터링 특성상 히트율이 올라가는 구조입니다.

LIID · Deep Dive

로컬 LLM · 촬영 요청까지 하나의 서비스로

로컬 LLM

위성 데이터는 외부로 반출할 수 없어 상용 LLM API를 쓸 수 없습니다. 그래서 LLM을 로컬 모델로 구성해 추론이 내부에서 끝나도록 하고, DEM·영상 정보와 결합해 3D 지형을 자연어로 조회·모니터링할 수 있게 설계했습니다.

SAR 촬영 요청 (Tasking)

사용자가 보고 있는 지역에 대해 SAR 영상 촬영을 직접 요청할 수 있도록 요청 접수 흐름을 서비스 안으로 넣었습니다. 조회 → 분석 → 신규 촬영 요청이 끊기지 않는 하나의 사용자 흐름이 됩니다.

Role

PM으로 요구사항·일정·이해관계자를 관리하면서 아키텍처 설계(서비스 경계, 캐시 계층 전략, 로컬 LLM 구성, SDPE 연계 구조)를 직접 담당하고 있습니다. 현재 진행 중인 프로젝트입니다.

Flagship Project 05

대용량 파일 전송 엔진 · 분산 파일 서버

Tech Lead · 설계 주도

OneDrive식 Windows API 전송 엔진 · 청크 병렬 전송 · k8s 오토스케일 · 큐 기반 부하 분산

Problem

기존에는 NAS에 권한을 부여하는 방식으로 파일을 다뤄, 대용량 전송이 느리고 서버 한 대에 부하가 몰렸습니다. 50GB급 데이터를 한 번에 밀어 넣으면 전송이 길어지고, 중간에 서버가 죽으면 처음부터 다시 올려야 했습니다. 트래픽이 몰릴 때 서버를 늘리거나 부하를 나눌 방법도 없었습니다.

Solution

NAS 권한 부여 대신 OneDrive처럼 Windows API 기반 전송 엔진을 직접 구축하고, 큰 파일은 청크로 나눠 병렬 전송해 50GB 데이터를 수 분 내에 옮기도록 설계했습니다. 파일 서버는 k8s 오토스케일업으로 부하에 따라 늘어나고, 큐 시스템으로 물리 서버 병렬 처리와 부하 분산을 구성. Redis 캐시로 전송 상태를 보관해 중간 서버가 죽어도 이어서 진행되게 했습니다.

청크 병렬 전송

50GB 파일을 한 스트림으로 밀지 않고 여러 청크로 쪼개 동시에 병렬 전송합니다. 대역폭을 최대로 활용해 수 분 내에 이동하고, 실패한 청크만 다시 보내면 되므로 재전송 비용도 최소화됩니다.

오토스케일 · 큐 부하 분산

파일 서버를 k8s 오토스케일업으로 구성해 부하가 오르면 Pod가 늘어나 트래픽을 나눠 받습니다. 그 앞에 큐 시스템을 두어 청크 작업을 물리 서버들에 고르게 분배, 특정 서버에 몰리지 않도록 병렬·분산 처리합니다.

Redis 캐시 · 전송 재개

전송 진행 상태를 Redis 캐시에 보관해, 처리 중이던 중간 서버가 죽어도 다른 서버가 마지막 지점부터 이어서 전송을 마칩니다. 대용량 전송 도중의 단일 장애가 전체 실패로 번지지 않습니다.

Role
